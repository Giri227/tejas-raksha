                       2. ARCHITECTURE & DESIGN
================================================================================

DESIGN PHILOSOPHY:
- Modular plugin-based architecture
- Async-first for high performance
- Non-destructive, read-only testing
- Polite crawling with rate limiting
- Extensible and maintainable

CORE COMPONENTS:

1. SCANNER CONTROLLER (src/scanner.py)
   - Orchestrates complete scan workflow
   - Manages three-phase execution:
     * Phase 1: Crawling (URL discovery)
     * Phase 2: Security checks (vulnerability detection)
     * Phase 3: Reporting (results generation)
   - Handles error recovery and logging
   - Provides real-time progress updates

2. CRAWLER MODULE (src/crawler/)
   - Fetcher: HTTP request handling with httpx
   - Parser: HTML parsing and link extraction with BeautifulSoup4
   - Frontier: URL queue management and deduplication
   - Controller: Crawl orchestration and depth management

3. SECURITY CHECKS (src/checks/)
   - Plugin-based architecture
   - Base class for consistent interface
   - Registry for automatic discovery
   - Context-based execution
   - 8 built-in security checks

4. REPORTER MODULE (src/reporter/)
   - HTML reporter with Jinja2 templates
   - JSON reporter for machine-readable output
   - CSV reporter for spreadsheet analysis
   - Generator for multi-format output

5. UTILITIES (src/utils/)
   - HTTP utilities and helpers
   - Rate limiter for polite crawling
   - Robots.txt parser
   - Input validators
   - Logging configuration

6. DATA MODELS (src/models/)
   - Finding: Individual vulnerability representation
   - ScanResult: Complete scan results
   - ScanConfig: Configuration management
   - ScanMetadata: Scan information and statistics

7. CLI INTERFACE (src/cli/)
   - Click-based command-line interface
   - Rich argument parsing
   - Configuration file support
   - Interactive prompts

DESIGN PATTERNS USED:
- Plugin Pattern: Security checks
- Strategy Pattern: Report generators
- Factory Pattern: Component initialization
- Observer Pattern: Progress logging
- Singleton Pattern: Configuration management

================================================================================
                    4. SECURITY CHECKS (8 BUILT-IN)
================================================================================

CHECK 1: EXPOSED SENSITIVE FILES
---------------------------------
Severity: HIGH
Category: Information Disclosure
Description: Detects publicly accessible sensitive files that should not be
             exposed, such as configuration files, backups, and version control.

Detected Files:
- .git/ directories and files
- .env environment files
- .htaccess Apache configuration
- config.php, database.php
- backup.sql, dump.sql
- phpinfo.php, info.php, test.php
- .DS_Store, Thumbs.db
- web.config, .htpasswd
- composer.json, package.json
- And 50+ more patterns

Detection Method:
- Probes common sensitive file paths
- Checks HTTP status codes (200 = exposed)
- Validates file content
- Reports exact URLs

Remediation:
- Remove sensitive files from web root
- Configure web server to deny access
- Use .gitignore properly
- Implement access controls


CHECK 2: DIRECTORY LISTING
---------------------------
Severity: MEDIUM
Category: Information Disclosure
Description: Identifies directories with directory listing enabled, which
             exposes file structure and potentially sensitive files.

Detection Method:
- Requests directory URLs
- Looks for directory listing patterns:
  * "Index of /" in title
  * "Parent Directory" links
  * Apache/nginx listing formats
- Identifies exposed directories

Remediation:
- Disable directory listing in web server
- Add index.html to directories
- Configure Options -Indexes (Apache)
- Configure autoindex off (nginx)


CHECK 3: SERVER HEADER ANALYSIS
--------------------------------
Severity: LOW to MEDIUM
Category: Information Disclosure
Description: Analyzes server headers for version disclosure and security
             misconfigurations.

Checked Headers:
- Server: Apache/2.4.41, nginx/1.18.0
- X-Powered-By: PHP/7.4.3
- X-AspNet-Version
- X-AspNetMvc-Version
- And other version-revealing headers

Detection Method:
- Examines HTTP response headers
- Identifies version numbers
- Flags detailed version disclosure

Remediation:
- Remove or obfuscate Server header
- Disable X-Powered-By header
- Use generic server tokens
- Keep software updated


CHECK 4: REFLECTED XSS (CROSS-SITE SCRIPTING)
----------------------------------------------
Severity: HIGH
Category: Injection
Description: Tests for reflected XSS vulnerabilities in GET parameters by
             injecting safe payloads and checking for reflection.

Test Payloads:
- <script>alert('XSS')</script>
- <img src=x onerror=alert('XSS')>
- <svg onload=alert('XSS')>
- <iframe src=javascript:alert('XSS')>
- And 10+ more payloads

Detection Method:
- Identifies forms and GET parameters
- Injects XSS payloads
- Checks if payload appears in response
- Validates reflection context

Remediation:
- Encode all user input in output
- Use Content-Security-Policy header
- Implement input validation
- Use framework auto-escaping
- Apply context-aware encoding


CHECK 5: ERROR DISCLOSURE
--------------------------
Severity: MEDIUM
Category: Information Disclosure
Description: Detects verbose error messages that reveal sensitive information
             about the application, database, or server configuration.

Detected Patterns:
- Stack traces (Python, PHP, Java, .NET)
- Database errors (MySQL, PostgreSQL, SQL Server)
- Framework errors (Django, Laravel, Spring)
- File path disclosure
- Configuration errors

Detection Method:
- Triggers errors with invalid input
- Scans response for error patterns
- Identifies framework signatures
- Reports error types

Remediation:
- Disable detailed errors in production
- Implement custom error pages
- Log errors server-side only
- Use generic error messages


CHECK 6: HTTPS CONFIGURATION
----------------------------
Severity: MEDIUM
Category: Cryptography
Description: Analyzes HTTPS configuration including TLS version, HSTS header,
             and mixed content issues.

Checks Performed:
- HTTPS availability
- TLS version (1.2+)
- HSTS header presence
- HSTS max-age value
- Mixed content (HTTP resources on HTTPS pages)
- Certificate validation

Detection Method:
- Tests HTTPS endpoint
- Examines TLS handshake
- Checks security headers
- Scans for HTTP resources

Remediation:
- Enforce HTTPS for all pages
- Enable HSTS with long max-age
- Use TLS 1.2 or higher
- Fix mixed content issues
- Redirect HTTP to HTTPS


CHECK 7: OPEN REDIRECT
----------------------
Severity: MEDIUM
Category: Injection
Description: Tests for open redirect vulnerabilities that allow attackers to
             redirect users to malicious sites.

Test Payloads:
- http://evil.com
- //evil.com
- /\\evil.com
- javascript:alert('XSS')
- And more bypass techniques

Detection Method:
- Identifies redirect parameters (url, redirect, next, return)
- Injects external URLs
- Checks Location header
- Validates redirect destination

Remediation:
- Whitelist allowed redirect destinations
- Validate redirect URLs
- Use relative URLs only
- Implement redirect tokens


CHECK 8: SECURITY HEADERS
--------------------------
Severity: LOW
Category: Security Misconfiguration
Description: Verifies presence and correct configuration of security-related
             HTTP headers.

Checked Headers:
- Content-Security-Policy (CSP)
- X-Frame-Options (clickjacking protection)
- X-Content-Type-Options (MIME sniffing)
- X-XSS-Protection (XSS filter)
- Referrer-Policy (referrer control)
- Strict-Transport-Security (HSTS)
- Permissions-Policy (feature control)

Detection Method:
- Examines HTTP response headers
- Checks for header presence
- Validates header values
- Reports missing/misconfigured headers

Remediation:
- Add all recommended security headers
- Configure CSP policy
- Enable X-Frame-Options: DENY
- Set X-Content-Type-Options: nosniff
- Implement HSTS with long max-age


CHECK 9: SQL INJECTION (ERROR-BASED) ⭐ NEW!
--------------------------------------------
Severity: HIGH
Category: Injection
Description: Detects SQL injection vulnerabilities through database error
             message analysis. Uses safe, non-destructive payloads.

Test Payloads:
- ' (single quote)
- '' (double single quote)
- 1' (numeric with quote)
- 1' OR '1'='1 (boolean-based)
- ' OR '1'='1' -- (comment-based)
- ' OR 1=1-- (numeric boolean)
- admin'-- (authentication bypass)
- 1' AND '1'='2 (false condition)

Detected Databases:
- MySQL (SQL syntax, mysql_ functions)
- PostgreSQL (PostgreSQL ERROR, pg_ functions)
- SQL Server (SQL Server Driver, mssql_ functions)
- Oracle (ORA-#### errors, oci_ functions)
- SQLite (SQLite.Exception, sqlite_ functions)
- Generic SQL errors

Detection Method:
- Tests GET parameters with SQL payloads
- Analyzes response for database errors
- Pattern matches error messages
- Identifies database type
- Reports vulnerable parameters

Error Patterns Detected:
- "SQL syntax.*MySQL"
- "PostgreSQL.*ERROR"
- "SQL Server.*Driver"
- "ORA-[0-9]{4}"
- "SQLite.Exception"
- "unclosed quotation mark"
- And 20+ more patterns

Remediation:
1. Use parameterized queries (prepared statements)
2. Never concatenate user input into SQL
3. Use ORM frameworks with parameterization
4. Validate and sanitize all input
5. Apply least privilege to database accounts
6. Disable detailed error messages in production

Example Fix (PHP with PDO):
  BAD:  $query = "SELECT * FROM users WHERE id = " . $_GET['id'];
  GOOD: $stmt = $pdo->prepare("SELECT * FROM users WHERE id = ?");
        $stmt->execute([$_GET['id']]);

PROGRAMMATIC USAGE:
-------------------

Python API Example:

from src.models.config import ScanConfig
from src.scanner import Scanner
import asyncio

async def main():
    # Create configuration
    config = ScanConfig(
        target_url="https://example.com",
        max_depth=2,
        concurrency=10,
        output_dir="./reports",
        report_formats=["html", "json"]
    )
    
    # Create and run scanner
    scanner = Scanner(config)
    result = await scanner.run()
    
    # Access results
    print(f"Found {len(result.findings)} vulnerabilities")
    for finding in result.findings:
        print(f"- {finding.severity.value}: {finding.title}")

# Run scan
asyncio.run(main())

                      9. CONFIGURATION OPTIONS
================================================================================

CLI OPTIONS (COMPLETE LIST):
----------------------------
-d, --depth INTEGER
  Maximum crawl depth (default: 2)
  Range: 1-10
  Example: -d 3

-c, --concurrency INTEGER
  Concurrent requests limit (default: 10)
  Range: 1-50
  Example: -c 20

--follow-subdomains
  Follow links to subdomains
  Default: False
  Example: --follow-subdomains

--delay-min FLOAT
  Minimum delay between requests in seconds (default: 0.5)
  Range: 0.1-10.0
  Example: --delay-min 1.0

--delay-max FLOAT
  Maximum delay between requests in seconds (default: 2.0)
  Range: 0.1-10.0
  Example: --delay-max 3.0

--no-robots
  Ignore robots.txt (use with caution)
  Default: False
  Example: --no-robots

--js-render
  Enable JavaScript rendering with Playwright
  Default: False
  Requires: playwright package
  Example: --js-render

--js-timeout INTEGER
  JavaScript rendering timeout in seconds (default: 5)
  Range: 1-30
  Example: --js-timeout 10

--auth-type [basic|digest]
  Authentication type
  Options: basic, digest
  Example: --auth-type basic

--auth-user TEXT
  Authentication username
  Example: --auth-user admin

--auth-pass TEXT
  Authentication password
  Example: --auth-pass secret123

-H, --header TEXT
  Custom HTTP header (can be used multiple times)
  Format: "Name: Value"
  Example: -H "Authorization: Bearer token"

-C, --cookie TEXT
  Custom cookie (can be used multiple times)
  Format: "name=value"
  Example: -C "session=abc123"

--timeout INTEGER
  Request timeout in seconds (default: 10)
  Range: 5-60
  Example: --timeout 15

-o, --output PATH
  Output directory for reports (default: ./reports)
  Example: -o ./my-scans

-f, --format [html|json|csv|all]
  Report format (can be used multiple times)
  Options: html, json, csv, all
  Default: html
  Example: -f html -f json

--enable-check TEXT
  Enable specific check(s) (can be used multiple times)
  Example: --enable-check XSSCheck

--disable-check TEXT
  Disable specific check(s) (can be used multiple times)
  Example: --disable-check DirectoryListingCheck

--config PATH
  Configuration file path (YAML or JSON)
  Example: --config config.yaml

-v, --verbose
  Enable verbose logging
  Default: False
  Example: -v

--no-warning
  Skip ethical usage warning
  Default: False
  Example: --no-warning

--help
  Show help message and exit
